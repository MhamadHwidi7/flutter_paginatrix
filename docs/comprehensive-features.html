üïê Clock System

Why Clock Abstraction?

The Clock system provides time abstraction for deterministic behavior in tests and reliable cache expiration:

üß™ Testable: Mock time in tests for predictable behavior
‚è∞ Reliable: Consistent time handling across platforms
üîÑ Cache Management: Precise control over cache expiration

Clock Interface

abstract class Clock {
  DateTime now();
  Duration elapsed(Duration duration);
  bool isExpired(DateTime timestamp, Duration ttl);
}

// Production implementation
class SystemClock implements Clock {
  @override
  DateTime now() => DateTime.now();
  
  @override
  Duration elapsed(Duration duration) => duration;
  
  @override
  bool isExpired(DateTime timestamp, Duration ttl) {
    return DateTime.now().difference(timestamp) > ttl;
  }
}

// Test implementation
class MockClock implements Clock {
  DateTime _currentTime = DateTime(2024, 1, 1);
  
  @override
  DateTime now() => _currentTime;
  
  void advance(Duration duration) {
    _currentTime = _currentTime.add(duration);
  }
}

Usage in Cache Management

class CacheEntry<T> {
  final T data;
  final DateTime timestamp;
  final Duration ttl;
  
  CacheEntry(this.data, this.timestamp, this.ttl);
  
  bool isExpired(Clock clock) {
    return clock.isExpired(timestamp, ttl);
  }
}

Testing with Mock Clock

test('cache expiration works correctly', () async {
  final mockClock = MockClock();
  final cacheStore = MemoryCacheStore<String, String>(
    clock: mockClock,
    ttl: Duration(minutes: 5),
  );
  
  // Add data
  await cacheStore.put('key', 'value');
  
  // Advance time by 6 minutes
  mockClock.advance(Duration(minutes: 6));
  
  // Data should be expired
  final result = await cacheStore.get('key');
  expect(result, isNull);
});

Automatic Retry Strategy

final retryPolicy = RetryPolicy(
  maxRetries: 3,
  baseDelay: Duration(seconds: 1),
  maxDelay: Duration(seconds: 30),
  backoffMultiplier: 2.0,
  jitter: true,
  retryCondition: (error) {
    return error is NetworkError || error is RateLimitError;
  },
);

Error Recovery Patterns

1. Automatic Retry: Network and rate limit errors are automatically retried
2. Fallback to Cache: Show cached data when network fails
3. User Notification: Clear error messages with retry options
4. Graceful Degradation: Partial functionality when some features fail
